# StadiumMind AI

**The AI Operating System for FIFA World Cup 2026 Stadiums**
Repository: [https://github.com/Code-Game-Ninja/Stadium-Mind.git](https://github.com/Code-Game-Ninja/Stadium-Mind.git)

        StadiumMind AI is a five-day FIFA World Cup 2026 GenAI hackathon build: a complete AI-enabled stadium operations platform for fans, volunteers, and organizers using Gemini, Firebase (Auth + Firestore), Socket.IO, real live-football and weather feeds, and a polished command-center interface.

        The `docs/` tree is the original engineering documentation; the working implementation lives in `apps/` and `packages/`.

        ## Hackathon Submission Details

        **1. Chosen Vertical**
        - **Stadium Operations, Fans, and Volunteers (Smart Venues)**: A comprehensive operating system designed for the FIFA World Cup 2026. It serves the three primary personas in a stadium: Fans (journey planning), Volunteers (SOP and incident resolution), and Organizers (command center decision-making).

        **2. Approach and Logic**
        - **Centralized Event-Driven Architecture**: The platform uses a Node.js backend with Socket.IO for real-time telemetry updates and Firebase (Firestore + Auth) as the single source of truth.
        - **Generative AI Integration**: Gemini 2.5 Flash is deeply integrated to power the core logic of the platform, transforming raw stadium data into natural language recommendations, creating personalized fan journeys, and acting as a live SOP assistant for volunteers.
        - **Proactive vs. Reactive**: Instead of just showing dashboards, the AI proactively analyzes simulated crowd telemetry and weather data to predict congestion and suggest actionable resolutions (with explainability and confidence scores) before incidents occur.

        **3. How the Solution Works**
        - **Fans (Match Hub & Journey Planner)**: Fans select their match, verify their ticket, and receive a personalized, step-by-step, gate-to-seat journey plan generated by AI based on their specific preferences and accessibility needs.
        - **Volunteers (Workspace)**: Volunteers log in to a dedicated portal where they can chat with an AI assistant loaded with stadium SOPs, report live incidents, and handle lost & found items.
        - **Organizers (Command Center)**: Organizers use a digital twin interface to monitor stadium health metrics, simulate crowd or weather events, and receive live AI recommendations for mitigating risks (e.g., redirecting crowds from a specific gate).

        **4. Assumptions Made**
        - **Simulated IoT / Crowd Data**: We assume the existence of stadium turnstile and density sensors. Because this real-world data is private and inaccessible, we built a robust simulation engine that generates realistic organic crowd telemetry over time.
        - **Simulated Ticketing System**: Fan ticket verification uses a predefined set of dummy tickets rather than connecting to a real FIFA ticketing backend.
        - **Centralized Command**: We assume organizers have the authority to act on AI suggestions (e.g., dispatching volunteers, closing gates) and that these actions can be pushed live through the Socket.IO network.

        ## Product Summary

        StadiumMind AI solves the fragmentation problem inside large tournament venues. Fans need personal routes and match-day plans. Volunteers need instant SOP help. Organizers need a live AI command center that turns crowd, transport, incident, sustainability, and volunteer data into recommended actions.

        The system uses:

        - A public **Match Hub** for schedule and stadium exploration.
        - **Ticket ID and QR simulation** for fan context without fan login.
        - A **Fan Journey Planner** that creates a gate-to-seat plan using match, ticket, crowd, preferences, and accessibility data.
        - A protected **Volunteer Workspace** using Firebase Auth and role-based access.
        - A protected **Organizer AI Command Center** with proactive alerts, what-if simulation, action history, incident timeline, confidence estimates, and explainability.
        - A **Gemini AI layer** with separate prompts for fan, volunteer, organizer, prediction, decision, reporting, and translation workflows.
        - A **live-data layer**: real football scores/lineups (Sportmonks), real stadium weather (Open-Meteo), and a simulated IoT sensor loop standing in for private crowd telemetry.

        ## MVP Scope

        The five-day MVP should include:

        - Match Hub with multiple simulated FIFA World Cup 2026 matches.
        - Ticket verification by ticket ID and QR simulation.
        - Fan journey planner with route, timeline, preferences, and accessibility.
        - Organizer command center with live stadium health, AI summary, proactive alerts, recommendations, confidence, explainability, incident timeline, action history, simulation controls, match brief, and end-match report.
        - Volunteer dashboard with SOP assistant, incident reporting, lost and found, and translation.
        - Firebase Auth for admin, volunteer, and fan roles.
        - Gemini API for all GenAI features.
        - Socket.IO powered realtime updates (snapshots + sensor heartbeat).

        ## Non-Goals For The Hackathon

        - Real FIFA ticket validation.
        - Paid voice AI.
        - Custom machine learning model training.
        - Real IoT hardware integrations.
        - Drone feeds or AR navigation.
        - Payment or ticket purchasing.
        - Production medical, security, or emergency dispatch authority.

        ## Key Design Principle

        Build a believable command center, not a feature list. Every feature should support one demo story: a fan gets from match selection to seat efficiently while the organizer prevents congestion using AI recommendations and a volunteer resolves incidents with SOP guidance.

### Environment variables

The app runs fully in **demo mode with no configuration** (in-memory store, deterministic AI, open routes). Add keys to unlock the real-data layers:

| Variable | App | Purpose | Default behavior if unset |
| --- | --- | --- | --- |
| `NEXT_PUBLIC_API_BASE_URL` | web | Backend REST base URL | `http://localhost:4000/api` |
| `NEXT_PUBLIC_SOCKET_URL` | web | Socket.IO URL | `http://localhost:4000` |
| `NEXT_PUBLIC_FIREBASE_*` | web | Firebase Auth + Firestore client config | Login screens unavailable |
| `PORT` | api | API port | `4000` |
| `GEMINI_API_KEY` | api | Gemini AI | Deterministic AI fallbacks |
| `GEMINI_MODEL` | api | Gemini model | `gemini-2.5-flash` |
| `GOOGLE_APPLICATION_CREDENTIALS` | api | Firebase Admin service account → Firestore persistence + API auth | In-memory store, open routes |
| `SPORTMONKS_API_KEY` | api | Real live-football feed (scores, minutes, lineups, events) | Manual Live Score Controller only |

Live weather needs no key — the API queries **Open-Meteo** at each stadium's real coordinates.

Copy `apps/web/.env.example` → `apps/web/.env.local` and `apps/api/.env.example` → `apps/api/.env` to configure. **Never commit the service-account JSON** (gitignored).

First-time Firebase setup: `cd apps/api && npx tsx src/seed.ts` seeds stadiums, matches, tickets, and the demo users below into Firestore + Firebase Auth.

### Demo path

1. **Match Hub** → pick France vs Spain → *I have a ticket*.
2. **Verify** `WC2026-453621` (or paste the QR payload).
3. **Journey planner** → choose arrival + preferences + accessibility → *Generate my journey* (map route + timeline + fan AI chat).
4. **Command Center** (`/organizer`, demo login) → health score, live metrics, AI summary, proactive alerts, digital twin.
5. **Simulation controls** → *Increase Crowd* / *Rain* / *Metro Delay* etc. → metrics, timeline, and a new AI recommendation update live over Socket.IO.
6. **Recommendation** → *Why?* (reason, confidence, expected outcome, signals) → *Apply* (mutates the twin + logs action history).
7. **Volunteer Workspace** (`/volunteer`, demo login) → SOP assistant, incident reporting, lost & found, translation.

### Runtime & data modes (read this)

Every layer is **real where a real source exists**; simulation is limited to what no hackathon team can legally obtain, and every panel badges its data source:

| Layer | Source | Real? |
| --- | --- | --- |
| Match scores, minutes, lineups, events | **Sportmonks live feed** (60s cache; organizer can take manual control, which then wins) | ✅ Real |
| Weather at each stadium | **Open-Meteo** at real coordinates (MetLife / AT&T / SoFi) — feeds metrics, health score, AI recommendations | ✅ Real |
| Persistence | **Firestore** — incidents, recommendations, actions, zones, volunteers, timeline survive restarts | ✅ Real |
| Auth & roles | **Firebase Auth** + server-side ID-token verification (admin/volunteer routes return 401/403) | ✅ Real |
| AI reasoning | **Gemini** server-side, deterministic fallback badged when offline | ✅ Real |
| Fan/volunteer/organizer actions | Real user-generated data over Socket.IO in real time | ✅ Real |
| Crowd/IoT telemetry | **Simulated sensor loop** — organic arrival-curve drift every 8s + scenario what-if injectors. Stadium turnstile feeds are private infrastructure; the ingestion interface (`apps/api/src/sensorLoop.ts`) is the swap-in point for a real feed | 🔶 Simulated by necessity |

- **Zero-config demo mode still works.** Without any keys the API runs in-memory with deterministic AI and open routes — nothing breaks.
- **Fan login is optional by design.** Fan context comes from match selection + demo ticket verification (ID or QR payload).

### AI layer

All Gemini calls run in the backend (`apps/api/src/ai.ts`) using the prompts in `prompts/`. Without `GEMINI_API_KEY`, deterministic fallbacks in `packages/shared/src/fallbacks.ts` produce equivalently-shaped output so the app never breaks. Responses badge whether Gemini or the offline fallback was used.

### Limitations

- Crowd/IoT telemetry is simulated (see table above) — real turnstile and density feeds are private stadium infrastructure. The sensor-loop module is the designed integration point.
- Ticket verification is against the demo ticket database, not a real FIFA system (surfaced everywhere in copy).
- Zero-config mode (no Firebase credentials) leaves API routes open and state in memory — configure Firebase for real auth + persistence.

### Deployment

The frontend and backend deploy to two different hosts because the API is a persistent Node process (Socket.IO rooms + background loops for the sensor tick and the proactive AI briefing), which serverless platforms like Vercel cannot keep alive between requests.

- **Web (`apps/web`) → Vercel.** Root `vercel.json` builds `@stadiummind/shared` then `@stadiummind/web` and serves `apps/web/.next`. Import the repo in Vercel, leave the root directory as the repo root (the `vercel.json` handles the workspace build), and set these Environment Variables in the Vercel project:
  - `NEXT_PUBLIC_API_BASE_URL` → `https://<your-render-api>.onrender.com/api`
  - `NEXT_PUBLIC_SOCKET_URL` → `https://<your-render-api>.onrender.com`
  - `NEXT_PUBLIC_FIREBASE_*` → same values as `apps/web/.env.example`

- **API (`apps/api`) → Render** (or Railway/Fly — anywhere that runs a persistent Node process). Root `render.yaml` defines the service: `npm install && npm run build --workspace @stadiummind/shared && npm run build --workspace @stadiummind/api`, started with `npm run start --workspace @stadiummind/api`, health-checked at `/api/health`. Set these Environment Variables on the Render service:
  - `WEB_ORIGIN` → `https://<your-app>.vercel.app` (so CORS + Socket.IO accept the deployed frontend)
  - `GEMINI_API_KEY`, `GEMINI_MODEL`, `SPORTMONKS_API_KEY`, `LIVE_FEED_MATCH_ID` → same as local `.env`
  - `FIREBASE_SERVICE_ACCOUNT_BASE64` → the whole service-account JSON, base64-encoded into one line (Render can't receive an arbitrary file the way local dev does). It's decoded to a temp file at boot; `GOOGLE_APPLICATION_CREDENTIALS` is set automatically from it. See `apps/api/.env.example` for the encode command.

Rotate `GEMINI_API_KEY` / `SPORTMONKS_API_KEY` before deploying — don't reuse whatever is sitting in a local `.env`. After both are live, point `npm run smoke` at the deployed API to re-verify before judging:

```bash
node scripts/smoke.mjs https://<your-render-api>.onrender.com/api
```
