        # Software Requirements Specification

        ## System Context

        StadiumMind AI is a web application with public and authenticated areas. The public area serves fans and general visitors. The authenticated areas serve volunteers and organizers. The backend owns data normalization, ticket verification, simulation state, AI orchestration, recommendation logging, and realtime event publishing.

        ## Actors

        - Fan: unauthenticated user with optional verified demo ticket context.
        - Visitor: unauthenticated user exploring match and venue information without a ticket.
        - Volunteer: authenticated user with volunteer role.
        - Organizer/Admin: authenticated user with admin role.
        - AI Service: Gemini-backed service invoked only by the backend.
        - Simulation Engine: deterministic event generator for crowd, transport, incident, and sustainability signals.

        ## Functional Requirements

        | ID | Requirement | Priority |
| --- | --- | --- |
| FR-001 | List simulated FIFA World Cup 2026 matches with stadium, city, time, teams, and status. | MVP |
| FR-002 | Verify ticket ID against demo ticket table and bind ticket to selected match. | MVP |
| FR-003 | Reject tickets that do not belong to the selected match and explain the mismatch. | MVP |
| FR-004 | Decode QR payloads containing ticket_id and optional checksum for demo verification. | MVP |
| FR-005 | Generate fan journey plans using ticket, preferences, accessibility, crowd, transport, and kickoff context. | MVP |
| FR-006 | Protect volunteer and admin routes using Supabase Auth and role metadata. | MVP |
| FR-007 | Publish stadium simulation updates through Socket.IO. | MVP |
| FR-008 | Generate organizer summaries, proactive alerts, recommendations, and explanations using Gemini. | MVP |
| FR-009 | Store recommendations and action outcomes in action history. | MVP |
| FR-010 | Create and update incidents from volunteer and admin surfaces. | MVP |
| FR-011 | Generate match brief before event and end-match report after event. | MVP |
| FR-012 | Provide deterministic fallback AI text when Gemini fails or quota is exhausted. | MVP |

        ## Non-Functional Requirements

        | Category | Requirement |
| --- | --- |
| Performance | Dashboard updates should render within 2 seconds of receiving a Socket.IO event. |
| Reliability | Demo mode must continue without Gemini by using static fallback responses. |
| Security | Admin and volunteer data must be role-protected by Supabase RLS and backend authorization. |
| Privacy | Do not require fan accounts. Ticket verification uses demo IDs and no real personal data. |
| Accessibility | Core flows must be keyboard reachable and readable on mobile. |
| Maintainability | AI prompts must be versioned in `prompts/` and response schemas in `schemas/`. |

        ## Data Requirements

        Data is modeled in Supabase Postgres. The implementation must include matches, stadiums, stadium zones, tickets, user profiles, volunteers, incidents, recommendations, action history, simulation events, AI logs, transport metrics, sustainability metrics, and fan feedback.

        ## AI Requirements

        AI output must be structured whenever it affects product state. Human-readable copy can be generated by Gemini, but actions, confidence, impacted zones, and expected outcome must be returned in JSON and validated. The backend must reject malformed AI responses and use fallback logic.

        ## Realtime Requirements

        Socket.IO events must support:

        - `stadium:snapshot`
        - `crowd:update`
        - `incident:new`
        - `recommendation:new`
        - `action:updated`
        - `simulation:scenario_applied`
        - `ai:summary_updated`
